# Слой: разбор AI-фоллоу-апов звонков

Битрикс24 после завершённого звонка генерирует **AI Follow-up**: транскрипцию, обзор встречи (тема, повестка, договорённости, action items), сегментированное саммари, инсайты по спикерам, оценку эффективности. Это готовый, чистый разбор — богаче и точнее сырой стенограммы. Слой забирает его с портала и заводит в заметки как разбор встречи (meeting-note нужного проекта).

По смыслу — это роль «Разборщик встреч» (`ROLES.md`), но источник не файл-стенограмма в `00-inbox/`, а API портала. Протокол разбора тот же; отличается источник.

**Режим — on-demand.** По расписанию не гоняется: звонки разбираются точечно, когда просишь, а не пачкой. В ежедневный прочёс чатов (`chat-sweep.md`) фоллоу-апы не входят.

Фундамент (настройка, безопасность, транспорт) — в `SKILL.md` рядом.

---

## Транспорт: особый путь (REST v3)

Методы `call.followup.list` / `call.followup.get` — это REST **v3**, не v1. Они живут по особому пути, и это главная ловушка транспорта. **Тот же вебхук их умеет** — отдельных кред не нужно, работает на уже настроенном `B24_WEBHOOK`. Единственное условие — у вебхука должен быть scope **`call`** (добавляется в правах входящего вебхука там же, где calendar/tasks).

> ⚠️ Форма пути ниже — **наблюдаемое поведение** на текущих порталах, добытое эмпирически, а не задокументированная гарантия. Роутинг v3-методов версионно-зависим и на твоём портале может отличаться. Если не заводится с первого раза — перебери формы (см. три варианта) или используй альтернативный транспорт внизу. Это не повод считать мост сломанным.

Наблюдение: **префикс `api/` идёт ПЕРЕД uid/token**:

- ✅ `POST https://<портал>/rest/api/<uid>/<token>/call.followup.<метод>` (тело — JSON) — срабатывает.
- ❌ `/rest/<uid>/<token>/call.followup.list` (с/без `api/` ПОСЛЕ токена) → обычно `ERROR_METHOD_NOT_FOUND` (v1-роутер этих методов не знает).
- ❌ `/rest/api/call.followup.list?auth=<токен-вебхука>` → обычно `ACCESSDENIED` (это форма OAuth-токена, не вебхука; не путать).

То есть берёшь свой `B24_WEBHOOK` (`https://<портал>/rest/<uid>/<token>/`) и вставляешь `api/` между `/rest/` и `<uid>`. Ошибки v3 приходят как `{error:{code,message}}`.

## Команды слоя (что дёргать)

- **Список фоллоу-апов** (`call.followup.list`) — звонки с **готовым** AI-разбором за период (звонки без AI-разбора не возвращаются). Курсорная пагинация под капотом; дата — в TZ владельца.
  - «что было на созвонах за неделю», «какие звонки с разбором за период» → список за N дней (обзор; сам разбор — по выбранным звонкам).
  - Фильтр по конкретному участнику работает только под админскими правами; на обычном вебхуке видны свои звонки (где ты участник или в связанном чате).
- **Полный разбор одного звонка** (`call.followup.get` по `callId`):
  - без транскрипта — всё, кроме тяжёлого транскрипта (обзор, саммари, инсайты, оценка, участники, action items). Хватает для большинства разборов.
  - с транскриптом — + полный транскрипт, нужен для точной атрибуции реплик и цитат.

Триггеры: «забери звонок `<callId>`» / «разбери фоллоу `<id>`» → сразу `get`. «Разбери последний созвон», «фоллоу-ап встречи с N» → сначала список, выбрать нужный `callId`, потом `get`.

## Протокол разбора (по одному звонку)

Источник чище стенограммы — обзор / action items / саммари уже выделены порталом. Но **проверка фактов, атрибуция и разбиение задач — на агенте**, не доверяй разметке вслепую.

1. **Получить материал.** Разбор звонка (с транскриптом, если нужна атрибуция/цитаты). Прочитать целиком, не по диагонали.
2. **План в чат ДО записи.** Перед любой записью в заметки покажи владельцу: в какой проект кладём, кого из участников как атрибутируем (полными именами из справочника людей), какие задачи чьи. Не перекладывай сверку имён на владельца вопросами постфактум.
3. **Люди.** Каждый участник → сверить со справочником людей (`02-people/`); имена в транскрипте — портальные display-names, в разбор и карточки выносить полными именами. Карточка на каждого упомянутого, не только на собеседника.
4. **Задачи — разбиение.** `actionItems` от портала — это **сырьё, а не готовый список твоих задач**. Задача собеседника → его карточка + открытые вопросы проекта. В свой личный слой (`my-focus`) — только то, что ты явно взял на себя.

> **Атрибуция по говорящему + адресату — сверять с транскриптом, не флипать.** Когда **ты — говорящий** и говоришь «*ты [сделай]*» — владелец задачи **адресат**; «*я [сделаю]*» — владелец **ты**. API-разметка может это перепутать (приписать поручение не тому) ИЛИ пропустить задачу, которую ты обронил голосом. Не доверяй и не отвергай атрибуцию API вслепую — резолвь говорящего и адресата по сегментам транскрипта.

5. **Цитаты.** Поля `quote` (в action items / договорённостях) и транскрипт — реальная речь, годятся как цитаты с атрибуцией. Без транскрипта/quote — это пересказ, без кавычек.
6. **Кросс-проектный скан.** Что из обсуждённого трогает другие активные проекты → раздел «Кросс-проектные следы» + апдейты соседних файлов.
7. **Записать meeting-note** по шаблону в `03-projects/<project>/meeting-notes/YYYY-MM-DD-тема.md` (регулярные 1-1 — в подпапке по человеку). Источник указать как «AI Follow-up звонка #`<callId>`, дата». Оценку встречи и инсайты спикеров — в психологический срез аккуратно, нейтрально; иронию/самооценку не превращать в факт состояния.
8. **Ценные находки — предложить, не файлить сами.** Наблюдение про человека, принцип, стратегический сигнал — предложить владельцу занести в постоянный слой.

## Хранение (zero-bloat)

- Сырой фоллоу-ап (транскрипт, JSON) в заметки **НЕ сохраняем** — он живёт на портале, перезапросить дёшево.
- Артефакт в заметках — только разобранный meeting-note (дистиллят).
- Нужна конкретная цитата/фрагмент позже — перезапросить разбор звонка, не держать архив сырья.

## Опциональные альтернативные транспорты (не нужны, пока работает вебхук)

Если по какой-то причине вебхук-путь недоступен, у методов фоллоу-апов есть и другие транспорты — но это продвинутое, для обычной настройки не требуется:
- OAuth локального приложения со scope `call` (транспорт с `?auth=<access_token>`, клиент рефрешит токен при 401).
- Кастомный серверный action на стороне портала (персональный hash, отдельная авторизация).

Любые такие креды (OAuth-токены, hash) — тот же класс секретов, что вебхук: только в локальном конфиге вне репозитория, никогда в заметки / чат / git.
