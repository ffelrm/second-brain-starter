# Слой: прочёс чатов всех видов и размеров

Самое содержательное на портале живёт не в полях объектов, а **в переписке**: кто кого блокирует, реальные сроки, договорённости, настроение команды. Этого нет в заметках. Этот слой — как забирать переписку всех видов: личку, группы, каналы, чаты задач, чаты-проекты, календарные. От «прочитай один чат» до «прочеши всё за месяц».

Фундамент (настройка, безопасность, транспорт, метод `im.dialog.messages.get`) — в `SKILL.md` рядом. Нужен scope `im` у вебхука (+ `disk` для файлов из чатов).

---

## Виды чатов — что ловим

Портал держит все диалоги в одном мессенджере. Один метод-сборщик — recent-хвост мессенджера (в текущих версиях портала это `im.v2.Recent.tail`; точное имя v2-метода свериться на своём портале — старые коробки могут отдавать recent иначе) — отдаёт хвост по **всем** разделам сразу:

| Вид | Что это |
|---|---|
| Личка (1-1) | Прямой диалог с человеком |
| Группа | Групповой чат |
| Канал | Канал-вещание |
| Рабочая группа | Чат рабочей группы/соцсети портала |
| Чат задачи | Обсуждение внутри задачи |
| Чат проекта (коллаб) | Проект, группирующий другие чаты (см. ниже) |
| Календарный | Чат события/совещания |
| CRM | Чат сделки |

Чего **не** видно: открытые линии клиентов (вне личного вебхука) и любые чаты, где ты не участник — by design, чужой контур.

## Три уровня: один чат → прочёс всего → прочёс за период

### 1. Прочитать один конкретный чат — `im.dialog.messages.get`

Когда назвали конкретный чат/диалог.
- Групповой/проектный чат — по его `chatId` (диалог `chat<chatId>`).
- Личка — по ID пользователя (диалог лички = id человека; резолв имени → ID через `user.get`, при однофамильцах уточни).
- Окно от новых к старым, есть потолок на запрос (~200) → глубже пагинация назад.
- Системные сообщения (создание чата, инвайты, смены сроков) — автор ID `0`, по умолчанию прячь.
- Вложения: список (имя / размер / автор / дата / disk-id) выводи в конце; содержимое конкретного файла тянется отдельно по disk-id (текст печатается, бинарь — в /tmp). URL скачивания не цитируй (auth-параметр).

### 2. Инкрементальный прочёс ВСЕГО — паттерн «collect»

«Что в чатах за вчера / за неделю», «что обсуждали» — забрать всё новое по всем чатам за период. Ключевые механики (без них прочёс тихо врёт):

- **Инкрементальный стейт.** Держи файл `{dialog_id: last_seen_msg_id}`. Каждый прогон берёт только сообщения с `id > last_seen`. Историческую массу не перечитываешь. Первый прогон заберёт всё за окно, дальше — только свежее. Так «прочёс с последнего раза» устойчив к пропущенным дням (выходные, отпуск): не теряются дни, добирается всё с последнего успешного прогона.
- **Закреплённые чаты не обрывают обход.** Pinned-чаты всплывают в начало recent-хвоста ВНЕ зависимости от даты (могут быть многолетней давности). Наивный «стоп на первом старом элементе» обрубит обход и спрячет свежую личку. Правильно: курсор пагинации и условие остановки считаются **только по незакреплённым**; закреплённые собираются, но обход не обрывают.
- **Батчинг + бюджет времени.** Чтение первой страницы N диалогов по одному — десятки round-trip'ов, на 50+ чатах не влезет в лимит вызова и упадёт БЕЗ вывода. Собирай work-list активных диалогов, читай нативным `batch` (≤50 подкоманд/запрос). Держи бюджет времени на весь прогон: при превышении — чекпоинт и чистый выход, `last_seen` НЕ двигаем → следующий прогон добирает. Крупный забор делится на 2 прогона, но НИКОГДА не падает по таймауту.
- **Читай только там, где реально новое.** Сверяй `messageId` recent-элемента с `last_seen` — тянешь тело только в диалогах с новыми сообщениями. Deep-обход не бьёт по API каждый прогон.
- **Исключения шумных чатов.** Мониторинг, CI-нотификации, статус-каналы — заведи список исключений: их не читаем, но стейт сохраняем (чтобы не потерять позицию, если вернём). Иначе шум забивает дайджест.
- **Файлы.** Текстовые ≤1 MB — можно тянуть содержимым сразу; крупнее — помечай «требует ручной докачки» (имя + disk-id); бинарные тихо пропускай (только имя + disk-id).

### 3. Прочёс за длинный период — backfill

«Прочеши все чаты/проекты за июнь», «обогати картину за квартал» — тяжёлый обход (все диалоги × вся история окна) не влезает в один вызов. Делай **резюмируемым**:
- Фаза A — перечислить объекты (все чаты / все проекты и их под-чаты).
- Фаза B — читать порциями, писать промежуточное в чекпоинт-файл (вне заметок, рабочий /tmp), выходить чисто.
- Повторяй тот же запуск, пока не напечатает `DONE`. Чекпоинт хранит прогресс — каждый запуск добирает следующую порцию.
- Raw в заметки НЕ кладём (zero-bloat) — это материал под синтез. Артефакт — свёртка/обогащение картины (что двигается по проектам, кто что ведёт, настроения, кросс-связи), а не сырые переписки.

## Чаты задач — где «сок» задачи

Обсуждение задачи содержательнее её полей. Живёт в **чате задачи**, не в легаси-комментах.
- У задачи есть `chatId`; сам чат — обычный диалог `chat<chatId>`, читается `im.dialog.messages.get`. Легаси-метод форумных комментов отстаёт (обрывается на старой дате) — чат полнее, включает мигрированную историю. Легаси — только фолбэк для редких задач без `chatId`.
- Системка — автор ID `0` (назначения, наблюдатели, смены сроков), по умолчанию прячь.

Два режима:
- **Одна задача** — весь тред задачи (когда назвали ID / «о чём в этой задаче»). Тело задачи (описание + статус + дедлайн) печатай над тредом.
- **Скан по человеку** — «что обсуждают у N в задачах за неделю». Список активных задач человека + окно чата по каждой, реплики режутся до дайджеста. **Для управленцев бери роль `member`** (наблюдатель), а не `responsible`: их собственные задачи-«на себе» обычно тихие/заметки, а «сок» — в задачах, где они постановщик/наблюдатель. Ограничивай число задач на скан (кап), полный текст — одиночным режимом по ID.

## Хвост своих задач — без обхода тысяч

Для **самого владельца** скан «по человеку с ролью member» не годится: у активного руководителя member-задач тысячи → таймаут. Вместо перебора — recent-хвост своего вебхука:
- Из recent берём только элементы типа «чат задачи» с активностью в окне — их десятки, не тысячи. Не падает.
- Дёшево (секунды): по каждому активному чату задачи — его последняя реплика (уже в recent, без доп. запроса). Видно, что шевелится + последняя строка.
- Глубже — дочитать полные треды окна (с капом на число чатов против таймаута).
- Новые задачи окна добираются отдельным узким запросом списка по `>=CREATED_DATE` (не весь список).
- Recent не строго по дате (закреплённые вклиниваются) — не обрывай пагинацию на первом «старом».

Это правильный инструмент для утреннего прочёса своих задач. Скан по роли member — для **подчинённых** (recent — хвост своего хука, чужой recent не возьмёт).

## Чаты-проекты (коллабы) — слой слепых пятен

В Битрикс24 есть фича «Проекты»: проект — это чат-**коллаб**, который **группирует** другие чаты (задачи, календарные синки, под-чаты). Содержания этих чатов обычно нет в заметках — прямой слой «слепых пятен».
- Список проектов — recent-элементы типа коллаб (по активности).
- Состав проекта — тот же метод с фильтром по родителю (`parentId` = id коллаба) → дочерние чаты.
- Чтение — `im.dialog.messages.get` по нужному чату (главному или дочернему).
- Свёртка по проекту — прочесать главный + все дочерние за окно, отдать по каждому живые (не системные) реплики.
- Тяжёлый периодный заход — тем же резюмируемым backfill, что выше.

Нюанс: список проектов = только коллабы. Новый **группчат/канал/личка** через него не виден — их лови общим прочёсом (раздел 2), помечая новые чаты.

## Карта покрытия — что прочёс видит и НЕ видит

Чтобы прочёс не создавал ложное чувство полноты, держи явную матрицу и проверяй её на каждом «прочёс выполнен»:

| Объект | Чем ловится | Статус |
|---|---|---|
| Личка (1-1) | общий прочёс | ✅ (проверяй ВСЕ 1-1 с активностью — там часто самый сок прямых договорённостей) |
| Группа / канал / рабочая группа | общий прочёс | ✅ |
| Чат задачи (ты участник) | общий прочёс + скан по человеку | ✅ (двойное) |
| Чат проекта-коллаба | свёртка проекта + общий прочёс | ✅ (двойное) |
| Календарный / CRM чат | общий прочёс | ✅ |
| Новая задача | узкий список по `CREATED_DATE` | ✅ |
| Новый чат любого типа | флаг «новый» в общем прочёсе (НЕ через список проектов) | ✅ |
| Разбор звонка (фоллоу-ап) | отдельный слой `followups.md` | ⚠️ on-demand, не в ежедневном прочёсе |
| Крупные вложения (>1 MB) | в прочёсе только имя + disk-id | ⚠️ докачка по disk-id вручную |
| Задача/чат, где тебя нет | — | ❌ by design (чужой контур) |

**Правило надёжности:** прежде чем что-то отбросить как неважное — кросс-канальная сверка. Описание+чат задачи → личка участника → фоллоу-ап звонка. Поручение часто рождается в личке, а не в самой задаче — если проверить только один канал, оно теряется.

## Границы и приватность

- Видно только диалоги, где ты участник.
- Содержимое чужих чатов и вложений в заметки **не файлить автоматически** — показывать в ответе, сохранять только по явной команде (и только дистиллят, не raw).
- Личные контакты сотрудников, чужие приватные события — не выносить в долговременное хранилище.
- Ссылки скачивания файлов не цитировать (auth-параметр).
