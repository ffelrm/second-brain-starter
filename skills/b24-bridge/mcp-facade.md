# Продвинутый путь: обернуть мост в локальный MCP-фасад

Базовый путь (`SKILL.md`) — агент сам собирает REST-запросы. Это работает с первой минуты и не требует ничего ставить. Но когда мост становится рабочей лошадкой — каждый день, по многу раз — у прямого пути вылезают четыре слабых места. Этот файл про то, как их закрыть, **не меняя ни одного метода портала**: тем же вебхуком, тем же REST, только с другим входом для агента.

Это **опционально**. Если дёргаешь портал раз в неделю — не строй, базового пути хватает.

---

## Идея фасада в одном абзаце

Между агентом и порталом ставится **локальный сервер-посредник** (MCP), который запускается на твоей машине и общается с агентом по stdio. Наружу он выставляет **бизнес-операции** («покажи задачи такого-то», «прочитай чат задачи»), а внутри держит всё техническое: URL вебхука, имена REST-методов, версии API, пагинацию, нормализацию. Агент больше не строит запросы — он вызывает инструмент и получает готовую структуру.

---

## Что это меняет

| Свойство | Прямой путь | Через фасад |
|---|---|---|
| Вызов | длинная команда с путями и флагами | `tasks_list(user_id=…)` |
| Ответ | текст/JSON под разбор | единый контракт `{items, count, limit}`, разметка вычищена, лимиты зажаты |
| Токен | в файле; на каждом вызове проходит через окружение агента | читается сервером; **рутинный путь его не касается** — см. оговорку ниже |
| Набор операций | любой метод портала | ровно то, что выставлено (**allowlist**) |
| Деструктивное | запрещено правилом | физически не выставлено наружу |
| Гочи | помнит агент | зашиты в сервер один раз |

---

## Шесть принципов, без которых фасад бессмысленен

1. **Allowlist, а не generic REST.** Не выставляй инструмент «вызови любой метод» — он возвращает все риски обратно. Наружу идёт конечный список бизнес-операций, и у каждой — внятное описание: по нему агент выбирает, что звать.
2. **Токен не покидает процесс.** Ни в ответах инструментов, ни в логах, ни в текстах ошибок. Логировать имя метода и тайминг — да; URL — никогда.

   > **Честная граница.** Это не абсолютный барьер: агент, у которого есть шелл, физически всё ещё может прочитать файл с секретом — фасад не отбирает у него такую возможность. Что он реально даёт: **рутинная работа перестаёт касаться токена**. Исчезает класс «случайно заэхоил секрет в вывод команды», а правило «не читай конфиг» из ежедневного требования превращается в редкое исключение. Разница между «этого не происходит по дисциплине» и «для этого больше нет повода».
3. **Read-only по умолчанию на уровне транспорта.** Клиент создаётся так, что write-методы отклоняются, даже если инструмент вызовет их по ошибке. Это второй барьер после allowlist.
4. **Единый контракт ответа.** Все списочные инструменты возвращают одну форму. Агент не гадает и не парсит разметку.
5. **Нормализация — в сервере.** Чистка HTML/BB-разметки, кламп лимитов (модель не должна уметь попросить 10 000 записей), приведение булевых флагов портала (одно поле приходит то строкой `"Y"`, то настоящим `true` — сравнение со строкой молча врёт).
6. **Ошибки — структурные.** Пробрасывай код ошибки портала как есть. «Что-то пошло не так» отнимает у агента возможность починиться (не тот scope? не тот метод? не та версия?).

---

## Главное решение: что НЕ класть в фасад

Ключевое ограничение: **вызов инструмента живёт внутри одного обращения агента.** Там есть потолок времени и негде держать чекпоинт. Поэтому распределяй по весу операции:

| Профиль | Куда | Почему |
|---|---|---|
| Интерактивное, секунды — «что у меня сегодня», «покажи задачи», «прочитай чат» | **Фасад** | Именно здесь выигрыш в эргономике и токенах |
| Тяжёлое, резюмируемое — прочёс всех чатов за неделю, отчёты за квартал, backfill за период | **Скрипт/CLI** | Переживает обрыв за счёт чекпоинтов; вызов инструмента — нет |
| Меняющее портал | **Фасад, но двухфазно** (ниже) | Нужен машинный барьер, а не обещание |

Не пытайся затащить в фасад всё. Инструмент, который стабильно не укладывается в лимит, — хуже, чем его отсутствие: он падает молча и без результата.

---

## Двухфазный write — барьер вместо обещания

Протокол «покажи превью → дождись подтверждения → выполни» держится на дисциплине агента. Машинная версия:

```
<действие>_preview(аргументы)  →  { превью, preview_id }   # портал НЕ тронут
<действие>_confirm(preview_id) →  выполнение               # только по валидному id
```

`preview_id` **рождается только в первой фазе**, живёт несколько минут и привязан к конкретному payload. Изменил текст после превью — старый id недействителен, нужна новая первая фаза.

Что это гарантирует, а что нет: механизм делает **невозможным одношаговое выполнение** — до отправки обязательно строится превью, и оно строится из того же payload, что уйдёт. Чего механизм не гарантирует — что человек это превью **прочитал**: агент может дёрнуть обе фазы подряд. Поэтому машинный барьер не отменяет правило из `write-path.md` (показать превью и дождаться явного «да») — он лишь убирает возможность обойти превью целиком.

## Что тестировать (без сети, на заглушках)

Четыре теста ловят почти все регрессии фасада:
1. **Список инструментов зафиксирован** — новый не появляется молча; меняешь набор — меняешь тест осознанно.
2. **Клиент создаётся read-only** — главный барьер не отключится незаметно.
3. **Контракт нормализации** — форма ответа, кламп лимитов, чистка разметки.
4. **Приведение флагов портала** — `"Y"` / `true` / `1` дают один результат.

---

## Стоимость, честно

- Появляется зависимость (SDK сервера) и шаг установки. Держи её **изолированной**: ядро, которое ходит в портал, должно оставаться работоспособным без неё — тогда фасад можно снести, не потеряв мост.
- Покрытие растёт вручную: каждый новый инструмент — это осознанное решение, а не «всё сразу».
- Два входа надо не расщепить: и фасад, и скрипты зовут **одно ядро**, иначе логика разъедется.

---

## Рядом: официальный справочник по API

У Битрикс24 есть отдельный MCP-сервер с **документацией** API (описания методов, параметры, допустимые значения). Данных портала он не отдаёт — это справочник, а не мост. Полезен ровно в одном месте: когда добавляешь новый слой и не хочешь угадывать точное имя метода или набор параметров — агент сверяется вместо догадки.

Две оговорки, прежде чем подключать:
- **«Без авторизации» ≠ «ничего не утекает».** Кредов ты действительно не отдаёшь, но наружу уходят сами запросы — а по тому, какие методы ты ищешь, читается, что ты строишь. Плюс ответы такого сервиса попадают в контекст агента как **недоверенный вход**: относись к ним как к документации из интернета, а не как к инструкции.
- **Адрес, состав инструментов и условия доступа меняются.** Актуальную точку подключения сверяй по текущей документации портала, а не по этому файлу — здесь она намеренно не зафиксирована, чтобы не протухла.
