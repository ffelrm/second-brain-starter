# Скилл: мост к Битрикс24 (фундамент)

**Задача.** Дать агенту свежие данные твоего портала Битрикс24 по запросу — календарь, задачи, оргструктуру, карточки людей, чаты, разборы звонков, отчёты — вместо того чтобы полагаться только на заметки, где данные могли устареть. Мост подключается один раз, дальше агент дёргает портал сам.

**Когда.** По запросам вроде «что у меня сегодня», «открытые задачи N», «свежая карточка человека», «что изменилось в оргструктуре», «что обсуждали в чате задачи», «разбери последний созвон», «что в проектах за неделю». А также — **при первом разворачивании системы**: если мост ещё не настроен, агент один раз предложит его подключить (см. `AGENT-RULES.md`, раздел «Мост к Битрикс24»).

Это **сенсор** — ускоритель, не обязательная часть. Базовая ценность второго мозга есть и без него. Но с ним агент видит живую правду портала, а не только зафиксированную в памяти.

**Этот файл — фундамент.** Настройка, безопасность, транспорт, базовые рецепты. Рабочие слои поверх фундамента — отдельными файлами рядом (см. «Рабочие слои» внизу).

---

## Модель в одном абзаце

Портал Битрикс24 отдаёт данные через **входящий вебхук** — персональную ссылку с токеном, которую ты выпускаешь в интерфейсе портала. Агент делает к ней обычные HTTP-запросы (через `curl`) и получает JSON. Токен вебхука — это секрет уровня пароля: кто им владеет, действует на портале от твоего имени в рамках выданных прав. Поэтому вся настройка сводится к одному: **выпустить вебхук, положить его в файл вне репозитория, дать агенту адрес этого файла.**

---

## Настройка (один раз, ~5 минут)

### 1. Выпусти входящий вебхук на портале

1. Зайди на свой портал под своей учёткой.
2. Открой раздел разработчика: **Приложения → Разработчикам → Другое → Входящий вебхук** (в разных версиях портала путь чуть отличается — ищи «Входящий вебхук» / «Inbound webhook»).
3. Отметь права доступа (scope). Полезный набор под все слои этого набора:
   - `user` — карточки людей,
   - `department` — оргструктура,
   - `calendar` — события,
   - `task`, `tasks_extended` — задачи,
   - `im` — чаты и сообщения (чтение обсуждений задач, лички, групп, проектов; прочёс),
   - `call` — AI-разборы завершённых звонков (фоллоу-апы),
   - `disk` — вложения (читать файлы из чатов, прикреплять к задачам).
   **Меньше прав — меньше риск при утечке.** Начни с `user/department/calendar/task/tasks_extended/im`, добавляй `call`/`disk` когда дойдёшь до этих слоёв. Не выдавай то, чем не пользуешься.
4. Сохрани. Портал покажет URL вида `https://<твой-портал>/rest/<число>/<длинный-токен>/`. Скопируй его целиком, **со слешем на конце**. `<твой-портал>` — это хост твоего портала как он есть: облачный (`компания.bitrix24.ru`, `.by`, `.kz`, `.de` и т.п.) или коробочный / self-hosted на своём домене либо IP. Мост работает с любым — важен сам URL вебхука, а не форма домена.

### 2. Положи вебхук в локальный конфиг (вне репозитория)

В корне системы уже есть папка `.b24-bridge/` с шаблоном. Скопируй шаблон в рабочий файл и впиши свой URL:

```bash
cp .b24-bridge/webhook.env.example .b24-bridge/webhook.env
chmod 600 .b24-bridge/webhook.env      # только владелец читает/пишет
```

Открой `.b24-bridge/webhook.env` в редакторе и вставь URL:

```ini
B24_WEBHOOK=https://<твой-портал>/rest/<число>/<токен>/
```

Файл `webhook.env` уже в `.gitignore` — он **никогда** не попадёт в git и не уедет в публичный репозиторий. `.example`-шаблон (без секрета) остаётся под версионностью как образец.

### 3. Скажи агенту, что мост готов

Например: «мост к порталу настроен, проверь связь». Агент дёрнет `user.current` и подтвердит, кто владелец хука. Дальше можно пользоваться.

## Безопасность (обязательно к соблюдению — и человеку, и агенту)

Токен вебхука = ключ к порталу от твоего имени. Правила жёсткие:

- **Секрет живёт только в `.b24-bridge/webhook.env`, права `600`.** Никогда — в заметках, в карточках людей, в разборах встреч, в истории чата с агентом, в git.
- **Агент не печатает вебхук.** Ни в выводе команд, ни при отладке. Для проверки настройки — `ls -la .b24-bridge/` (видно, что файл есть и права 600), **никогда** `cat .b24-bridge/webhook.env` (вывод bash попадает в чат = утечка). Не включать трассировку (`set -x`) в командах, которые касаются URL.
- **Read-only по умолчанию.** Запросы, которые только читают (`*.get`, `*.list`, `user.current`), — свободно. Действия, меняющие портал (создать задачу, отправить сообщение), — только по протоколу подтверждения (см. `write-path.md`).
- **Права вебхука — по минимуму.** Не выдан scope — метод недоступен, и это правильно. Расширяй точечно.
- **Компрометация → ротация.** Заподозрил, что токен где-то всплыл (скопировал в заметку, показал на экране, закоммитил) — сразу выпусти новый вебхук на портале, старый удали, обнови только `webhook.env`. Токен в git-истории считается скомпрометированным, даже если файл потом удалён.
- **Держи `.b24-bridge/` вне файл-синков.** Если папку системы синхронизируешь (Obsidian Sync, облачный диск, телефон) — добавь `.b24-bridge/` в исключения синка, чтобы токен не разъехался по устройствам.
- **Приватность вывода.** Если запрос вернул то, чего не должно быть в долговременном хранилище (личные контакты сотрудников, чужие приватные события календаря) — не сохранять это в заметки.

## Транспорт: как агент зовёт портал

Базовый вызов — POST к методу REST. Метод и параметры подставляются по месту; вебхук читается из конфига, но **не эхоится**:

```bash
# Подгрузить вебхук в переменную окружения текущего процесса (молча, без вывода URL):
set +x
export $(grep -E '^B24_WEBHOOK=' .b24-bridge/webhook.env)

# Пример вызова: кто владелец хука
curl -s -X POST "${B24_WEBHOOK}user.current.json" | python3 -m json.tool
```

- Тело метода — `application/x-www-form-urlencoded`. Параметры Битрикс24 разворачиваются в форму `filter[KEY]=value`, `select[0]=ID` и т.п. С `curl` их удобно передавать через `-d`, например `-d 'filter[RESPONSIBLE_ID]=<id>' -d 'select[0]=ID' -d 'select[1]=TITLE'`.
- Ответ — JSON. Полезная нагрузка в поле `result`; постраничные методы возвращают `next`/`total` — листай до конца.
- На ошибку метод возвращает `{ "error": ..., "error_description": ... }`. Частые причины: не тот scope у вебхука, опечатка в имени метода, битый токен.
- **Большие списки на крупном портале — всегда с фильтром.** Методы вроде «перечислить все хранилища / все задачи всех людей» без фильтра перебирают тысячи объектов и висят. Ограничивай по владельцу / отделу / дате.
- **Пакетные вызовы.** Когда нужно дёрнуть один метод по многим объектам (прочитать 50 чатов) — используй нативный `batch` (до 50 подкоманд в одном HTTP-запросе). Десятки round-trip'ов схлопываются в единицы — критично, если агент работает с лимитом времени на один вызов.

Агент может обернуть это как угодно (одноразовый `curl`, короткий `python3`-скрипт со `stdlib`). Держи механику простой и переносимой — никаких тяжёлых зависимостей ради моста.

> **Два пути реализации.** Всё описанное выше — **путь А**: агент сам собирает запросы. Ноль зависимостей, любой метод доступен, работает с первой минуты. Когда мост становится ежедневным инструментом, у него вылезает цена: длинные команды в каждом вызове, токен защищён правилом а не барьером, гочи держатся в памяти агента. Тогда есть **путь Б** — обернуть тот же мост в локальный MCP-фасад: агент вызывает бизнес-операции, а токен, имена методов и нормализация остаются внутри сервера. Да, это добавляет зависимость — но её держат изолированной, так что ядро, ходящее в портал, продолжает работать и без неё (фасад можно снести, не потеряв мост). Чертёж, принципы и что тестировать — в `mcp-facade.md` рядом; там же ссылка на готовую реализацию, если собирать самому не хочется. Путь Б опционален: дёргаешь портал раз в неделю — оставайся на А.

> **Про лимит времени на вызов.** Если агент работает в песочнице с потолком на один вызов (например ~45 секунд) и без фоновых процессов, переживающих вызов, — тяжёлые прогоны (обход сотен чатов, отчёты за квартал) делай **резюмируемыми**: пиши промежуточный результат в чекпоинт-файл и выходи чисто, следующий запуск добирает остаток. Не один долгий вызов. Паттерн — в `chat-sweep.md` (backfill).

## Базовые рецепты

Смысловые операции поверх транспорта. ID методов — стандартные REST Битрикс24.

### Человек — `user.get` / `user.current`
Резолв человека по ID, email или части имени → должность, отдел, контакты. **Всегда ищи человека по имени/почте до обращения по ID:** в крупной компании легко перепутать однофамильцев (несколько людей с одним именем). Нашёл ID — дальше по нему.

### Календарь — `calendar.event.get`
События владельца за день/неделю (`from`/`to` + тип владельца). Ключевое — отличить **встречу** от **личного блока**. На каждое событие смотри поля:
- `is_meeting` — главный флаг: есть приглашённые (встреча) или одиночное событие в своём календаре (личный блок).
- `attendees` — список участников со статусами (хост / принял / отказ / ждёт); `host_id`/`host_name` — кто создал.

Классификация: событие с приглашёнными → встреча, нужна подготовка. Одиночный блок с бытовым именем (обед, спорт, врач, занятие) → личное. Одиночный блок с **проектным** именем («поговорить по проектам», «синк по X») → не классифицируй молча, **спроси владельца, с кем это** — фактически может быть встречей, созванной вне портала. (Тонкость контракта: для участника-не-владельца портал иногда отдаёт флаг встречи как булев `true`, а не строку — нормализуй, иначе митинг покажется личным блоком.)

### Задачи — `tasks.task.list` / `tasks.task.get`
Ключевой нюанс руководителя: почти все «видимые» задачи — чужие, где ты постановщик (`CREATED_BY`) или наблюдатель (`AUDITOR`). Роли фильтра: `responsible` (висит лично на исполнителе), `originator` (ты поставил), `auditor` (наблюдаешь).
- Для вопроса «что висит» — `responsible`.
- Для **утреннего выноса** фильтруй жёстче: дедлайн в ближайшие дни **и** `CREATED_BY ≠ RESPONSIBLE_ID` (исключить «заметки сам себе»).
- Тело задачи (`DESCRIPTION`) и статус/дедлайн — отдельные поля в `tasks.task.get`; сам «сок» задачи — в её **чате обсуждения**, не в полях (см. `chat-sweep.md`).

### Оргструктура — `department.get`
Дерево отделов с руководителями (`PARENT_ID` даёт цепочку подчинения, `HEAD_ID` — руководитель отдела). Полезно для «свежести»: сравнить живую структуру портала с локальной копией в заметках и отдать расхождения человеку на решение — **автоматически заметки не переписывать** (парсер упрощённый, в заметках много контекста, которого в портале нет).

### Чтение любого чата — `im.dialog.messages.get`
Базовый метод чтения диалога: чат задачи, личка с человеком, групповой чат. Окно идёт от новых к старым (есть потолок на запрос, глубже — пагинация). Системные сообщения (назначения, смены сроков) имеют автора с ID `0` — фильтруй, если нужен только живой разговор. Ссылки на скачивание файлов не цитируй — они могут нести auth-параметр. Полный слой чтения чатов всех видов — в `chat-sweep.md`.

## Границы (общие для всех слоёв)

- **Видно только то, где ты участник** — свои задачи, чаты, события и своё поддерево подчинения. Чужие приватные данные мост не достаёт (и не должен).
- **Свежие данные не отменяют заметки.** Портал даёт актуальные контакты/должность/расписание; накопленный контекст (наблюдения, паттерны, история отношений) — из заметок. Ценность — в сумме.
- **Zero-bloat.** Сырьё (транскрипты, выгрузки, переписки) в долговременное хранилище не кладём — перезапрос дёшев. В заметки идёт только дистиллят (свёртка, meeting-note).
- **Корпоративный портал — не end-to-end.** Приватный диалог закрыт от коллег, но у администратора портала есть доступ к логам мессенджера. Для приватности от самой компании это не тот канал.
