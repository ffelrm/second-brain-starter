# Слой: write-путь (действия, меняющие портал)

Всё, что **меняет** портал — создать задачу, отправить сообщение, прикрепить файл. Отдельно от read-слоёв, потому что ставка выше: сообщение уходит **от твоего имени** (вебхук под твоей учёткой, не бот), задача реально создаётся у человека. Поэтому весь слой — под жёстким протоколом подтверждения.

Фундамент (настройка, безопасность) — в `SKILL.md` рядом.

---

## Жёсткий протокол (для любого write-действия)

**Агент не выполняет write сам.** Всегда:

1. Человек просит явно («поставь задачу N по теме X», «напиши N в личку: …»).
2. Агент **резолвит получателя** через `user.get` (при однофамильцах — уточнить ID, не угадывать).
3. Агент запускает действие в режиме **превью** (dry-run) — портал НЕ трогается, показывается что именно уйдёт: кому (имя + ID), текст, вложения.
4. Агент показывает превью человеку **в чате** одним блоком.
5. Человек подтверждает или правит («да, отправляй» / «отмени» / «измени текст»).
6. Только после явного «да» — реальный вызов. Вернуть ID созданного объекта.

Не запускай реальный вызов сам, без шага 5. Не выполняй «по умолчанию».

> **Совет ≠ команда.** Если в разговоре ты (владелец) обронил «надо бы написать N» / «стоит поставить задачу» — это **обсуждение**, а не команда на отправку. Агент выполняет write только на явную инструкцию, не на пересказ намерения.

---

## Создать задачу — `tasks.task.add`

Превью → подтверждение → создание. Возвращается ID задачи.

**Прикрепить файл** (нужен scope `disk`):
- Файл сначала грузится на Диск (`disk.folder.uploadfile`), потом привязывается к задаче через поле файлов задачи (хранит ID объекта на Диске, не сам файл).
- Загрузка идёт **до** создания задачи — если файл не загрузился, задача не создаётся (нет половинчатого результата).
- Гоча: метод листинга хранилищ Диска на крупном портале **всегда с фильтром** по личному хранилищу владельца — без фильтра он перечисляет тысячи хранилищ подчинённых и висит дольше лимита запроса.

## Отправить сообщение — `im.message.add`

Приватный канал «не через задачу»: в задачу всегда может дописаться наблюдатель, а личный диалог двух людей закрыт. Та же команда шлёт и в личку, и в групповой/проектный чат — разница в адресате:

- **Личка** — адресат = ID пользователя (приватный диалог один-на-один).
- **Группа / проект** — адресат = `chat<id>` (сообщение видят **все** участники, не приватно). В группе можно **тегнуть** человека упоминанием-BBCode `[USER=<id>]Имя[/USER]` (ID — через `user.get`) — отрендерится кликабельным пингом.

Файлы уходят нативным вложением (грузятся на Диск → коммитятся в диалог), не голой ссылкой.

**Оговорка приватности** (проговаривать, если тема чувствительная): приватно ОТ КОЛЛЕГ — никто не вклинится в личный диалог. Но корпоративный портал — не end-to-end: у администратора есть доступ к логам мессенджера. Для приватности от самой компании это не тот канал.

---

## Про лимит времени на вызов

Если агент работает в песочнице с потолком на один вызов и без переживающих вызов фоновых процессов — write-операция должна уложиться в **один** вызов (загрузка файла + создание за раз). Долгие фоновые прогоны с поллингом лога там не работают — не рассчитывай на них для write.
