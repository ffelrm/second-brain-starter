# Слой: рабочие отчёты сотрудников

Регулярная свёртка рабочих отчётов сотрудников (модуль учёта рабочего времени Битрикс24) в дайджест. Полезно руководителю с большим деревом подчинения: линейный персонал и руководители нижнего уровня сдают периодические отчёты, из них собирается **пульс низов** направления — темы, настроение, темп, ранние сигналы, которые иначе до тебя не доходят.

Это слой **инсайтов с низов**, а не дисциплинарный контур. Низкое покрытие (сдаёт только часть дерева) — это состав тех, кто реально обязан отчитываться, а не «провал дисциплины». Не помечай как тревогу.

Фундамент (настройка, безопасность) — в `SKILL.md` рядом.

---

## Транспорт

Рабочие отчёты обычно отдаются отдельным серверным методом (в разных установках — свой; на облаке это может требовать отдельной авторизации помимо основного вебхука). Практический смысл для агента одинаков: есть источник, который по окну дат возвращает отчёты сотрудников дерева + справочник (кто в каком отделе, кто чей руководитель). Если у твоего портала такого слоя нет или он недоступен — этот файл можно пропустить, на остальные слои моста он не влияет.

Любая доп-авторизация (персональный hash / ключ этого метода) — тот же класс секретов, что вебхук: только в локальном конфиге вне репозитория, никогда в заметки / чат / git / вывод.

---

## ⚠️ Главная гоча: «полное вхождение периода в окно»

**Это самое важное в слое.** API отдаёт только те отчёты, чей период **полностью внутри** запрошенного окна (`DATE_FROM ≥ окно.от` И `DATE_TO ≤ окно.до`). **Пересечение не работает.** Значит одно окно-фильтр НЕ покрывает все форматы отчётов сразу.

А форматов у людей несколько, и они сосуществуют:
- **Недельный** (ISO-неделя, Пн–Вс).
- **«Зарплатный» месяц** со сдвигом (например 25-го по 24-е) — чтобы успеть посчитать до конца календарного.
- **Календарный месяц** (1–31).
- **Плавающий месячный** ~28–31 день с **произвольным днём старта** (например кадэнс 27→26). Это **главная ловушка**: ни одно якорное окно (ISO / зарплатное / календарное) его не объемлет, а API отдаёт только отчёты целиком внутри окна — поэтому такой отчёт выпадает из всех «ровных» прогонов.
- **Короткие custom-периоды** (3–25 дней) — у новичков, при переходах, у уволенных.

**Вывод:** никогда не доверяй одному ровному окну (неделя / месяц / зарплатный месяц) для **полного** охвата — оно даст частичный срез под один формат.

## Как забрать полностью

Для полного охвата за период — запрашивай **широким диапазоном дат** (больше 31 дня) и разбивай его на скользящие окна:
- Скользящее окно ~31 день с шагом 1 день, **с забегом старта на 31 день назад** от начала диапазона (чтобы окна заходили ниже `date_from` и ловили плавающий период, который стартовал раньше, но пересекает диапазон), + фильтр-обрезка результата обратно к запрошенному диапазону + дедуп.
- Это гарантированно ловит любой ≤31-дневный отчёт, пересекающий диапазон, при любом дне старта — включая плавающие.
- Цена — медленнее (round-trip-bound, десятки секунд на пару месяцев). Для быстрого «примерно» можно ровный якорный срез (одно окно формата), но помни: плавающие при этом пропадут.

Порядок величин (иллюстрация на большом дереве за ~5 месяцев): перебор по календарным месяцам дал примерно треть отчётов; календарные + зарплатные месяцы — около двух третей; широкий диапазон со скользящим окном — все. То есть «ровные» фильтры теряли заметную долю. Отсюда правило: длинный период → широкий диапазон, не перебор ровных окон вручную.

Ровное окно одного формата (одна неделя / один месяц / один зарплатный месяц) — только когда нужен **ровно один якорный формат быстро**, а не полнота.

## Точечные пуллы

- «Что сделал N на неделе» → по одному человеку за неделю (быстрый якорный путь).
- «Забери ВСЕ отчёты N за период» / месячные / непонятного формата → тот же **широкий диапазон** по конкретному человеку (ловит и плавающие). Не перебирать зарплатные/недельные вручную, не гадать день старта.
- По отделу / за один день — аналогично, сужая фильтр.

Накапливай замеченные **кадэнсы по людям** (кто пишет плавающим 27→26, кто с задержкой задним числом) — это ускоряет точечные пуллы.

## Дайджест — линза «пульс низов»

Цель — не «кто сдал / не сдал», а **темы / настроение / темп** снизу. Правила формата:

- **Главная единица — тема, а не человек.** «Тема X (N упоминаний: имена) — что говорят» лучше, чем перечисление «A сделал X, B сделал X».
- **Считай повторения.** Тема у 2+ человек → в первую секцию (пульс). Одиночное — в фактологический срез внизу.
- **Цитаты обязательны** для настроения/темпа — только дословно из текста отчёта, без реконструкции. Настроение без прямой речи не передаётся.
- **Темп и нагрузка — отдельная секция.** Маркеры: «не успеваю», «продуктивно», «много вопросов», «упёрлись», «спокойная неделя». Считай частоту и тон. Одновременный «завал» у нескольких — сигнал, вынести отдельно.
- **Иерархия (кто под кем) — техническая сноска в конце**, не структурирующая секция.
- **Аномалии** — пустые отчёты у обычно подробных, резкая смена фокуса, новый неизвестный руководитель в данных → выписать, предложить проверить оргструктуру.

Артефакт — один файл на период (`04-journal/work-reports/YYYY-Wnn.md` или аналог). Тренды/динамика — когда накопится база из нескольких периодов.

## Что НЕ выносить в дайджест

- **Искажённые расшифровки имён** — нормализуй сразу (сверяй со справочником людей), не тащи «как услышалось».
- **Список задач без интерпретации** («закрыл 12, открыл 8») — это статистика, не сводка; помечай «отчёт пустой / только задачи», не растягивай.
- **Прямые цитаты** — только если в отчёте дословно; никаких реконструкций.
- **Финансовые числа** — только точные или математически выводимые, никогда «прикидка».
- **Самооценки / ирония** — не вытаскивать как факт состояния.
- **Премачурные ярлыки** — не называть разовое наблюдение «принципом» / «сдвигом» / «прорывом».
- **Карточки людей автоматически из отчётов не обновляем** — отчёт это срез за период, не вечный материал. Ценный паттерн про человека — отдельным пунктом владельцу, он решит, переносить ли.

## Хранение и границы

- **Zero-bloat.** Raw-отчёты в заметки не кладём; для страховки на время свёртки можно кэш в /tmp (mode 600), после подтверждения — выкинуть. Нужна цитата позже — перезапросить по конкретному человеку.
- **Граничные случаи:** пустой результат — валидный успех («отчётов за период нет»), не повторяй как ошибку. Нет head-доступа у ключа — сообщи, что источник сгенерирован не от головы дерева. HTML в полях отчёта (`<br>` и т.п.) — очистить перед записью.
- **Приватность.** Дайджест — управленческий контекст (темы/настроение), не персональные данные. Личные контакты, HR-чувствительное (выплаты, найм) — не в общий артефакт, только отдельным предложением владельцу.
