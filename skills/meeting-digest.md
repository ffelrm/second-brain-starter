# Скилл: разбор встречи (стенограммы)

**Задача.** Превратить сырую стенограмму встречи в структурированное знание внутри системы — так, чтобы через месяц ты нашёл это поиском и понял без перечитывания оригинала. Работает **глубоко и по одной встрече за раз**: цель — качество, не скорость.

**Когда.** Автоматически — когда «Архивариус» встретил в `00-inbox/` файл, похожий на стенограмму (таймкоды + длинный диалог), он делегирует разбор сюда, не пытаясь уложить в общий inbox-сценарий. По команде — «разбери эту встречу / стенограмму `<файл>`».

Это развёрнутая версия роли «Разборщик встреч» из `ROLES.md` — там короткий промпт, здесь полный протокол. Тот же протокол применяется к AI-разборам звонков с портала (см. `b24-bridge/followups.md`) — отличается только источник (файл vs API).

---

## Порядок работы

### 1. Прочитать стенограмму целиком
Не по диагонали. Длинный файл — читай чанками, но без пропусков. Короткое саммари в начале, если есть, — ориентир, не замена чтению.

### 2. Определить контекст (люди + проект) ДО разбора
- **Проект.** К чему относится встреча? Нет такого проекта — отметь, предложишь создать в плане.
- **Люди.** Сопоставь **всех** упомянутых со справочником людей (`02-people/`). Расшифровка коверкает имена стандартно (одна фамилия слышится как другая) — прогоняй через поиск по справочнику, **не перекладывай сверку на владельца** вопросами «а кто это». Владельцу выноси только те имена, которых в справочнике реально нет, явной формулировкой «не нашёл, проверь». Если подключён мост к порталу — сверь спорное имя через живой портал до вывода «не нашёл». Нет даты в имени файла — не ставь дату загрузки наугад, восстанови по календарю (там же список участников для сверки).

**Общие микрофоны.** Расшифровка часто отдаёт реплики нескольких людей одному «спикеру» — тому, через чей микрофон шёл звук в переговорке. Если одному спикеру приписана длинная реплика, где явно меняются смысловые блоки (тема, регистр, технический слой) — **не атрибутируй вслепую**. Выдели блоки с таймкодами и спроси владельца, кто какой вёл. До ответа не разноси реплики по людям, не делай выводов об их позициях, не обновляй карточки. Схему микрофонов зафиксируй в метаданных разбора (`shared_microphones`).

**Многолюдные встречи (планёрки, стратсессии, 5+ спикеров из разных направлений).** Не фильтруй контент по «своей зоне» при формировании разбора — фильтр зоны применяется **только к action items** (что лично на тебе). Сам разбор — **полный фактологический срез**: блок на каждого спикера (2-3 строки фактов минимум, молчавших пропускаем), значимые операционные/финансовые дельты как контекст, **все** объявленные решения (включая корпоративные программы «сверху», даже если тебя не назначили — это среда, в которой ты работаешь).

### 3. Извлечь слои знания
Для каждого — короткая формулировка сути + таймкод, не пересказ реплик.

**Базовые девять:**
- **a) Решения** — что договорились делать/не делать (включая «решили подождать» и «решили не делать»).
- **b) Открытые вопросы** — что осталось нерешённым. Для каждого: чья сторона (я / собеседник / обоюдно) + **триггер возврата**. Лови формулировки откладывания («вернёмся позже», «поговорю при случае», «переспим с идеей») — это не итог, а сигнал, что вопрос уходит в «когда-нибудь». Обязательство возврата на тебе (ты это сказал), но без явного триггера оно теряется. Триггер — конкретная дата / событие («после релиза») / «при следующем синке с N». Не находится — попроси владельца выбрать.
- **c) Action items** — кто, что, к какому сроку. Нет срока — «срок не зафиксирован», не додумывай. **Задача собеседника НЕ идёт в твой личный слой** (`my-focus`) — она в разборе + в карточке собеседника. В личный слой едет только то, что ты явно взял на себя.
- **d) Разногласия** — где сказали разное. Ценно само по себе, даже если не разрешилось.
- **e) Новые объекты** — люди, продукты, компании, термины, которых ещё нет в системе.
- **f) Психологический срез** — коротко по каждому: состояние, заметные триггеры, стиль, признаки перегрузки/напряжения. **Минимально, для ранней диагностики, не психоанализ.** Только то, что реально проявилось, без догадок «мне кажется, он чувствует». Детали — в карточку человека, здесь выжимка.
- **g) Кросс-проектные следы** — что трогает другие активные проекты. Отдельно — люди, формально в одном проекте, оперативно в другом (натяжения). Каждая связь — строкой, с указанием, какой файл/вопрос меняется.
- **h) Твои принципы и паттерны** — что проявилось как управленческий принцип («я всегда стараюсь…», «если вижу X — делаю Y»). Коротко + цитата-опора → в `principles.md` со статусом «черновик», флаг «уточнить формулировку».
- **i) Цитаты** — не больше 5, только где формулировка сильнее пересказа. С таймкодом и именем.

**Плюс две сильные секции:**
- **j) Зеркало на себя** — прогон встречи против твоих формализованных точек роста (открытые вопросы про себя, зоны роста из operating-manual, активные принципы). По каждой проявившейся: где сработало, где слабее, где не было повода. **Мало говорил / не вёл встречу — это тоже наблюдение**, не пробел: «вёл X, ты включился по Y, в Z подсветил рамку». Цель — абсолютная прокачка (работаем с каждым часом материала), не сравнение «эта планёрка против прошлой».
- **k) Что нового про собеседников** — по каждому: повторение известного паттерна (с отметкой «N-й раз того же триггера»), новый стиль, динамика против карточки. Паттерн повторился второй раз → обобщение в раздел «Паттерны и тактика» карточки, не только эпизод в разборе.

### 4. Контр-эвристика: отдельный проход ПОСЛЕ первого
Первый проход ловит явное. Второй — систематические пропуски, которые проскакивают одной репликой посреди технического обсуждения:
- **Сигналы-обмолвки** — короткие реплики, маскирующие открытый вопрос: «потом подумаем», «надо будет посмотреть», «возможно, туда придём». Каждая = открытый вопрос с триггером.
- **Атрибуции задач в чужих репликах** — «фикс на X», «возьми это, N», «отдай это W» — попадают в карточку соответствующего человека, даже если задача не в твоей зоне.
- **Архитектурные сдвиги** — «X становится Y», «расширяем функцию», «открываем наружу», «переходим на». Фиксируются как сигналы, даже если мимоходом в чужом блоке.
- **Операционные риски в технических блоках** — «модель не работает», «откат», «экономика не сходится», «нагрузка непредсказуемая». В открытые вопросы проекта, не теряются как «детали».
- **Сигналы рынка / партнёров** — реальное использование продукта клиентами, потребности партнёров, реакции рынка. Материал для продуктовой/BizDev-линзы.
- **Программы «сверху»** — когда руководство объявляет программу/проект, даже не назначая тебя. В решения встречи как контекст среды.

### 5. Что НЕ выносить (мусор для разбора)
- Антураж: технические проблемы со связью, погода, шутки, анекдоты.
- Корпоративы, бронирования, мелкие HR-операции (если не сигнал).
- Личное самочувствие участников.
- Технические детали без операционного риска (название библиотеки, версия пакета — если из этого не вытекает риск).
- Рутинные ритуальные отчёты «всё идёт нормально» без значимой дельты.

**Тест для каждого блока:** «попадает под категорию выше? Если нет — выноси».

**Конфиденциальное.** Всплыл факт, которым владелец явно не делится (скрытые кадровые решения, подготовка замен, увольнения) → в `01-self/confidential.md` с полным контекстом и списком мест, где нельзя писать. В разборе и карточке — нейтральная формулировка без намёков.

### 6. Показать план переноса ДО действий
Не абстрактное «разнесу по файлам», а конкретно: «файл X → раздел Y → такая запись». Владелец подтверждает/правит — **только потом** исполняешь.

### 7. Целевые файлы
- **Главный артефакт** — разбор в `03-projects/<project>/meeting-notes/YYYY-MM-DD-тема.md` (регулярные 1-1 — в подпапке по человеку, чтобы история лежала кучей). Это source of truth по встрече.
- `03-projects/<project>/open-questions.md` — проектные открытые вопросы (чья сторона).
- `03-projects/<project>/my-focus.md` — только action items **лично на тебе** + решения ближайших двух недель. Личный слой, пиши аккуратно. **Задачи собеседника сюда не попадают.**
- `01-self/open-questions.md` — только стратегические личные вопросы, не всё подряд.
- `01-self/principles.md` — принципы-кандидаты (черновик) + цитата.
- `02-people/<человек>.md` — карточки на **всех упомянутых**, не только собеседника. Есть карточка — дополняй, **никогда не перезаписывай**.

### 8. После подтверждения
Исходник стенограммы — в архив (`99-archive/`): цитата или атрибуция реплики могут понадобиться, а перезапросить файл нельзя. Удаляй только если уверен, что перепроверять не будешь. Обнови индекс, если появились новые файлы.

---

## Правила
- Имена — полными, как в справочнике людей; нет записи — предложи завести в плане.
- Таймкоды — в формате оригинала.
- Русский (или твой рабочий язык); термины не переводи через силу.
- **Не хвали встречу** («продуктивно обсудили») — пиши факты.
- Стенограмма путаная, не уверен в интерпретации — **скажи, где именно сомнения**. Лучше три честных «не понял» в плане, чем десять выдуманных решений.
- Не додумывай action items, которых не было. Лучше пусто, чем вода.
